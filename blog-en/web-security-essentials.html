🚀 A Security-Dev Accelerator: Five AI Prompt Templates

🛡️ Security-check prompts for developers

1. Code Security Audit
Prompt: I have a piece of [Node.js/PHP/Python] code that handles database queries [paste code]. Check for SQL Injection or unauthorized-access risks, and provide an optimized version using parameterized queries.

2. CSP 3.0 Header Generator
Prompt: Generate a strict Content-Security-Policy 3.0 header. Requirements: allow scripts only from the local origin and [your CDN domain], disable unsafe-inline, and set a report-to endpoint.

3. Auth Logic Audit
Prompt: Review the following login logic: [describe flow, e.g., using JWT stored in LocalStorage]. Point out the potential risk under an XSS attack and provide an implementation recommendation switching to HttpOnly cookies plus a CSRF token.

4. Encryption Best Practices
Prompt: I need to store users' national ID numbers. Recommend a 2026-preferred symmetric encryption algorithm (such as AES-256-GCM) and explain how to manage keys (KMS) without exposing the encryption key.

5. Rate Limiting Strategy
Prompt: Write a rate-limiting middleware for my Nuxt 3 / Nitro backend. Requirements: limit a single IP to 60 requests per minute, and apply stricter defense to the login endpoint to prevent brute-force attacks.

Three Layers of Defense: Protect Your Site From Modern Threats

1. Transport layer: HSTS beyond HTTPS

HTTPS alone isn't enough. You should enable HSTS (HTTP Strict Transport Security). This tells the browser that, for the next year, the domain may only be accessed over an encrypted connection — completely shutting down downgrade attacks.

2. Browser layer: kill XSS with CSP 3.0

Content Security Policy (CSP) is the nuclear weapon against XSS. In 2026, we recommend the 'nonce' (unique random value) strategy: even if an AI-generated attack script is successfully injected, it can't execute without the correct random token.

3. Server layer: zero trust and least privilege

Assume your server could be breached at any moment. Reduce database account privileges to the minimum (for example, Select only, never Drop), and enforce authentication on every API request. This ensures that even if one component fails, the whole castle still stands.
